Subject: Proposal and Prototype for CTCAE v4 Publish Process

Attachments: CTCAEQuickReference.pdf; ctcae.quickRef.draft.zip
Hi to all,
In fulfillment of VCDE responsibilities, I was responsible for prototyping and proposing a potential process for publishing CTCAE v4 (and future revisions).  I have reached a point where I would like to share the preliminary results.
 
To develop the process, I focused on creating an updated version of the CTCAE quick reference, which is the hardcopy that I received.  Code was developed in Java to generate documents for publication.
 
I believe the general publication process is in keeping with prior discussions, and goes as follows:
  • Develop CTCAE either in LexWIKI or OWL.  It doesn't matter which, as long as the end result can be imported to a LexBIG/EVS repository.  There are also certain properties expected by the document generation code (e.g. 'Grade 1', 'Grade 2', 'Short Name').  The current names were taken from the LexWIKI-derived source, with the expectation that OWL source would use similar naming conventions.
  • Load CTCAE source to the LexBIG/EVS repository.
  • Run the Java program 'org.LexGrid.LexBIG.util.ctcae.QuickRefPublisher'.  All source and dependencies for this program are released and available in svn (links provided below).  Note that the program currently uses the direct Java API to talk to the LexBIG repository, but could be modified to run against the caGrid API if necessary.  We only need query capability.
  • When launched, the program will present a menu of all code systems loaded in the LexBIG/EVS repository.  The user needs to select CTCAE from the list.
  • The program generates PDF, HTML, and RTF versions of the document.  Each of these has advantages and may be worth considering (more on this below).  It takes about 5 seconds to run.
  • To get a feel for what is generated, please see the attached PDF.  Note that this is not being submitted for review of the content at this point.  It is possible that some of the content that I had is not in final form.  The intent is to give you a feel for how the content could be packaged and submitted for review (or everyday use) by end users.
So that's the basics... here are some details ...
 
The CTCAE quick reference is divided roughly into 4 sections: front cover, introduction, body (event tables), and back cover.  The table of contents could be considered a fifth section.  Here are the approaches taken for each piece of the puzzle...
  • Front cover: The cover is auto-generated by taking some 'stock' wording and inserting the version number of the CTCAE selected by the user.  I spent some time futzing to get things to look like the hard copy, perhaps too long, but I felt it was important to ensure that we could deliver something that looked reasonably close to the v3.0 publication.  Overall dimensions for the document were also kept as faithful as possible to the original (~5" x 7").
  • Introduction: This is the first page beyond the cover, which provides a narrative overview of the document contents.  It is also tricky to automate, since the content is not generated.  Instead of generating it on the fly, the program looks for a standard rtf file with this content.  This allows the rtf to be edited through a standard word processor such as MS Word, WordPad, or Open Office.  I think this makes the most sense, since it will allow the authors to revise the wording using familiar tools without needing to dive into code.
  • Once this intro page is edited, it must be exported to PDF and PNG versions for use by the program.  This is a manual step.
  • There are free programs that enable this (e.g. PDF995 can print to PDF, and many programs such as Windows Paint can be used to take a snapshot for export to PNG).  The exported size is recommended to be ISO B6 dimensions (~5" x 7", to match the target document).  The program takes it from there.
  • Event Tables: These are automatically generated by querying the vocabulary through LexBIG/EVS.  Note that the draft does not contain annotations such as 'Remark' or 'Also Consider'.  They are not currently in our repository.  In the future, the program could by modified to recognize and deal with them.  An alternative is to edit the generated RTF afterward to include additional not in the repository itself (see comments on RTF below).
  • Back Cover: The HHS, NIH, and NCI Logos and document barcode are inserted to this page.  Each of these is located in a separate graphic file that is imported and dynamically inserted to the document, so a logo or barcode can change without requiring code change.
  • Table of Contents: Note here...  this was previously covered in the Introduction section.  It could be added there again via manual edits (as described in the Introduction section above).  However, the author would need to see a generated document first to know what page numbers to enter.  As an alternative, 'chapter' references are now generated automatically, and can be used in PDF viewers (as PDF Bookmarks) or RTF navigation (e.g. view the 'Document Map' in Word).
A few thoughts as well on the various formats...
  • PDF: This is the primary format that I was targeting originally.  PDF is nice in terms of being portable across OS platforms, carries a relatively light footprint, and allows the publisher a great deal of control over what the end user can do with it..  The thought here is that it may make sense to distribute CTCAE in a downloadable format rather than hardcopy.  If so, PDF would be a good choice.  It would also be a good choice for making printouts.
  • There is a lot of power here potentially for the future.  By default, I generated PDFs with AES 128 bit encryption.  Encryption can help prevent unwanted tampering (e.g. someone trying to alter and distribute as if it came from NCI).  I also enabled permissions for printing, introducing annotations/comments, screen readers, but no encryption of metadata.  You can password-protect to control access to various capabilities by user or owner, etc.
  • There is a method in the publish program where this can all be customized (I take no credit here, it was provided to me by the API that I relied on, named iText).
  • HTML: In addition to redistribution and print, it seemed likely that NCI might want to serve a standard web page (html) with the CTCAE content.  This is also generated by the program.  The output is not quite as polished as the PDF, but it is functional and could be prettied up further by those with web-guru-ish capabilities.
  • RTF: This is also generated by the program.  While not recommended as a redistribution format or for serving on the web, it does offer the advantage of being editable by standard office tools.  So, basically, if the program doesn't generate something quite right it provides an intermediate format that authors can modify without requiring changes to the program itself.  While we would like the program to be turn-key and generate everything 100% correct right off the bat, reality doesn't always cooperate.  It is nice to have an edit option.
  • The generated RTF, like the HTML, isn't quite as pretty as the PDF.  However, I think it is passable as a starting point for applying edit and polish.
One additional note...
  • You might notice that the 'Introduction' section in the generated RTF (and HTML) is inserted as an image instead of editable text.  This may seem confusing, since RTF is intended to be editable, and in my description of the 'Introduction' section above I said we start from RTF source.  To make a long story short, it was too messy to try and take the RTF from one source and insert it into another programmatically.  Because of formatting mismatches, automated insertion of text from one RTF file to another can start to look like junk rather quickly.  So by default I inserted the image.  However, if an author really wants to edit the intro material in the RTF output, they can manually copy/paste the content from one file to the other and be on their way.
 
So there it is.  I propose adhering to the above process moving forward with V4 publication, but would be happy to discuss further as necessary.  I have attached copies of the pdf, html, and rtf output (unaltered) for comment.  All source files are available in the NCI Subversion repository under the "LexEVS" GForge project.  The direct URL is at https://gforge.nci.nih.gov/svnroot/lexevs/contrib/trunk/ctcae_publish.  I will add the above to a readme for documentation.
 
Please have a look and let me know how you would like to proceed from here.
