LexBIG Data QA System - Domain Model

Documentation relating to the domain model used by LexBIG Data QA System.

[ overview | details | references ]

Overview

The LexBIG Data QA System attempts to validate individual data states and transformations by using a combination of counts and semantic rules.  To organize the counting scheme and rule implementation, we have chosen to define items to count and semantic rules by domain model object.  By using a unified conceptual domain model as the basis for all QA across the various life cycle stages, we are able to ensure that the representation of this conceptual model remains consistent as data moves its way through the system.

Details

The LexBIG Data QA System domain model is based on the basic LexGrid data model.  Following are the content domain objects that we attempt to account for.

[... from the LexBIG Programmer's Guide and Mayo's LexGrid Model pages...]
  • codedEntry
  • comment
  • definition
  • instruction
  • presentation
  • property
  • propertyQualification (not a LexGRID object, modeled on NameAndValue LexBIG object)
  • propertyLink (may be like a relationship, e.g. acronymOf)
  • associationTarget
  • associationData
  • associationQualification
Following are the metadata domain objects in our reference model
  • Association (object property, like UMLS REL/RELA)
  • AssociationQualifier (like UMLS ATN)
  • CodingScheme (singleton)
  • Hierarchy (which association represents a hierarchy, like UMLS RELA)
  • Language (of presentations, like UMLS LAT)
  • Property (like UMLS ATN)
  • PropertyLink (like UMLS REL/RELA)
  • PropertyQualifier (like UMLS ATN)
  • RepresentationalForm (same as UMLS TTY)
  • Source (like UMLS SAB)
Following are the history domain objects in our reference model
  • ChangeEvent
